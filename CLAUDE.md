# hook-marketplace — Claude Context

## What This Is

An open-source hook-first marketplace for Claude Code — a CLI package manager (`hookpm`), a public registry, a security certification layer, and an enterprise private registry. The goal: make sharing Claude Code hooks as easy as `npm install`.

- **Repo:** private, on this machine at `taakshal/hook-marketplace/`
- **Research & strategy docs:** `docs/` — read these before planning any feature

---

## The Product (Three Layers)

```
Layer 1: hookpm CLI (TypeScript, open source, free forever)
          hookpm install / remove / list / search / publish / verify

Layer 2: Public registry + author platform (free)
          hook.json schema + index.json + security pipeline
          author accounts, dashboard, download stats, engagement activities
          hook claiming, ratings, leaderboards, featured hooks

Layer 3: Certification + enterprise (paid, when marketplace has critical mass)
          certified badge ($99–$299/yr) — manual security review
          enterprise private registry ($299–$2k/mo) — SSO/SAML, audit logs, air-gapped
```

**Revenue model:** Start with 100% free — CLI, registry, author platform, all features. Build supply (authors) and demand (users) first. Introduce paid tiers only when the marketplace has enough critical mass that certification has real value.

## Tech Stack

| Layer | Choice |
|-------|--------|
| CLI | TypeScript + Node.js, published to npm as `hookpm` |
| Registry API | Hono (TypeScript, runs on Cloudflare Workers) |
| Database | Supabase (Postgres) |
| Object storage | Cloudflare R2 (zero egress fees) |
| Auth | Clerk (GitHub OAuth for authors) / API keys (programmatic) |
| Docs + web UI | Starlight (Astro-based) on Cloudflare Pages |
| Testing | Vitest |
| Deploy | Cloudflare Workers (API) + Cloudflare Pages (docs/UI) |
| Signing | minisign / Ed25519 |
| Phase 1A registry | GitHub repo + GitHub Actions (zero infra cost) |

**Note:** Fly.io is not used. Cloudflare Workers replaces it — same Hono code, runs at the edge, generous free tier. Zero migration cost if we ever move.

## Build Phases

```
Phase 1A — GitHub-backed CLI (current)
  hookpm install/remove/list/search via GitHub raw URLs
  Authors submit hooks via Pull Request
  No accounts, no server
  Cost: $0

Phase 1B — Author platform (next, still $0)
  hookpm publish → Cloudflare Workers API → R2 + Supabase
  Author accounts via Clerk (GitHub OAuth)
  Author dashboard (Cloudflare Pages)
  Download tracking, ratings, hook claiming
  Author engagement: streaks, leaderboards, featured hooks, community votes

Phase 2 — Certification + paid tiers (when marketplace has critical mass)

  Certified badge system
  Stripe payments ($99–$299/yr certification, $299–$2k/mo enterprise)
  Enterprise private registry (SSO/SAML, audit logs, air-gapped)
```

**Strategy:** Free everything first. Author engagement builds the supply side. Certification becomes desirable only when the badge has real value — authors pay because the marketplace already has users.

## Design-First Workflow (non-negotiable)

All work in this repo follows a **design-first, test-driven** process. No code is written before a design exists.

### Design Documents

- Every feature, subsystem, and architectural decision is documented in `docs/design/` as a Markdown file
- File naming: `YYYY-MM-DD-<topic>.md` (e.g. `2026-03-10-scaffold.md`)
- Each design doc must include:
  - Purpose and scope
  - Architecture / data flow with **Mermaid diagrams**
  - Component breakdown
  - Interface contracts (types, API shapes, CLI flags)
  - Security considerations
  - Open questions / trade-offs
- Design docs are **living documents** — updated in sync with code changes
- When code diverges from design (intentionally), update the design doc first, then change the code

### Workflow Order

```
Design doc written → design-reviewer (Opus) reviews → approved → RED tests → Code → Design doc updated if needed
```

1. **Design first, end to end** — complete the full design before any coding begins
2. **Opus reviews every design doc** — `design-reviewer` agent runs on every new or revised design doc; 🔴 Critical blocks implementation
3. **Every design doc has a concise TL;DR at the top** — ≤5 sentences: what, why, which phase
4. **Design drives code** — code must match the design doc; never the reverse
5. **Test-driven implementation** — RED tests written before implementation code
6. **Design updates** — if implementation reveals a design flaw, update the doc before changing the code

### Rules

- Do not write implementation code without an approved design doc in `docs/design/`
- Do not skip the design phase for "small" changes — scope creep starts with exceptions
- Design docs are committed alongside the code they describe
- When a design changes, the old design is not deleted — add a `## Revision History` section

## The `hook.json` Schema (source of truth)

Every hook in the registry must have a valid `hook.json`. The schema lives in `packages/schema/src/schema.ts` and is the contract for the entire registry. **Never bypass schema validation.**

Key fields: `name`, `version`, `description`, `author`, `event`, `matcher`, `handler`, `permissions`, `capabilities`, `tags`, `security` (registry-populated, not author-populated).

Full spec: `docs/deep/technical-spec.md`

---

## `settings.json` Merge Rules (critical)

The hardest part of `hookpm install`. Rules that must never break:
- Always atomic write (temp file → rename, never direct overwrite)
- Never clobber existing hooks in the same event array
- Merge by event type, not replace
- Write lockfile (`hookpm.lock`) after every install/remove
- `--prepend` flag for security hooks that must run first
- Always validate `settings.json` is parseable before and after write

Merge algorithm in `docs/deep/technical-spec.md`. Tests in `packages/cli/src/settings/__tests__/`.

---

## Security Rules (non-negotiable)

This project's attack surface is unusual: hooks run shell commands with Claude agent permissions. Security is a product feature, not a compliance checkbox.

- **Every hook submission gets automated static analysis** before merging to registry
- **Certified hooks get manual security review** — the `security-reviewer` agent runs on every PR that touches `registry/hooks/`
- **`security-reviewer` blocks merge** on any 🔴 Critical finding — no exceptions
- **Signature verification** must be tested on every CLI release
- **CVE-2025-59536 and CVE-2026-21852 patterns** are in the static analysis checklist — check for shared-repo injection and API token exfiltration patterns specifically
- **Never store API keys or credentials** in registry hooks or in any file

## Git Workflow

### Branch Structure

```
main          ← protected, source of truth — PR only, no direct push, no force-push ever
develop       ← integration branch, all feature PRs target here
feature/*     ← feature branches (feature/cli-install-command)
bug/*         ← bug fixes (bug/settings-merge-clobber)
security/*    ← security fixes (security/cve-scan-update) — fast-track review
```

### Merge Flow

```
feature/* or bug/*  →  develop  →  main
security/*          →  main (fast-track, skip develop for critical fixes)
```

### Merge Strategy

- Always rebase before PR: `git rebase develop`
- `feature/* → develop` — rebase merge (`gh pr merge --rebase`)
- `develop → main` — fast-forward only: `git merge develop --ff-only`

### Commit Format

```
type(scope): concise imperative summary

Optional body — explain the WHY, not the WHAT.
```

| Type | Use for |
|------|---------|
| `feat` | New production code — command, route, component |
| `fix` | Bug fix in existing production code |
| `security` | Security fix or security infrastructure improvement |
| `test` | Any test file — RED or GREEN phase |
| `docs` | CLAUDE.md, API docs, strategy docs, README |
| `schema` | `hook.json` schema changes |
| `chore` | Config, tooling, package.json, CI |
| `refactor` | Code change with no behavior change |
| `perf` | Performance improvement |

| Scope | Covers |
|-------|--------|
| `cli` | `packages/cli/` — all CLI commands |
| `settings` | `packages/cli/src/settings/` — settings.json merge |
| `schema` | `packages/schema/` — hook.json schema |
| `registry` | `registry/` — hook files, index, submission |
| `security` | Signing, verification, static analysis, CVE scanning |
| `api` | `api/` — hosted registry API |
| `db` | Database schema and migrations |
| `docs` | `docs/` |
| `infra` | Deployment, CI/CD, GitHub Actions |
| `agents` | `.claude/agents/` |

### Commit Rules

- **Never add `Co-Authored-By` or any co-authorship trailer**
- Never mention Claude, AI tools, or any co-author in commit messages
- One commit = one concern — if you need "and", split it
- Never commit a broken state — every commit must pass `typecheck` and `lint`
- Schema changes are always their own commit
- Security fixes always get a `security` type, never hidden in `chore`

### TDD Commit Sequence

Every feature produces commits in this order — no skipping:

```
schema(scope): <if schema changed>
test(scope): add RED tests for <feature>        ← qa-engineer
feat(scope): implement <feature>               ← coder
test(scope): add integration tests             ← qa-engineer GREEN phase
docs(scope): document <feature>                ← doc-writer
```

## Agents

| Agent | Model | When to call |
|-------|-------|-------------|
| `design-reviewer` | **Opus** | After every design doc is written or revised — before implementation begins. Blocks on 🔴 Critical. |
| `workflow-orchestrator` | Haiku | First — outputs exact execution plan for any task |
| `web-searcher` | Haiku | Gather external info — npm APIs, Claude Code changelog, competitor updates |
| `codebase-reader` | Haiku | Map relevant code before planning any feature |
| `planner-analyser` | Opus | Brainstorm + write experiment or feature plans |
| `coder` | Sonnet | Implement from a plan — dispatch multiple in parallel for independent scopes |
| `qa-engineer` | Sonnet | RED tests before implementation, GREEN verification after |
| `code-reviewer` | Sonnet | Review after every implementation step |
| `security-reviewer` | Sonnet | **Mandatory on every hook submission, every CLI security feature, every auth change** |
| `schema-validator` | Haiku | Before and after any `hook.json` schema change |
| `cli-tester` | Sonnet | End-to-end CLI behavior testing — install/remove/settings.json correctness |
| `doc-writer` | Haiku | Code docs, API docs, session learnings |

### Parallel vs Sequential

- **Parallel (background):** `web-searcher`, `codebase-reader`, `code-reviewer`, `security-reviewer`, `schema-validator` — read-only
- **Sequential (must wait):** `planner-analyser` → `qa-engineer` (RED) → `coder` → `qa-engineer` (GREEN) + reviewers
- **TDD is mandatory:** No `coder` starts until `qa-engineer` has written and confirmed failing RED tests
- **Security is mandatory:** `security-reviewer` runs on every hook submission and every auth/CLI security PR — never skipped

### Token Management

- One specific task per agent — not one broad topic
- Pass file paths between agents, not raw text
- Save agent outputs to `docs/` — agents read from files, not from conversation

## Do NOT

- Do not push directly to `main` — always via PR
- Do not merge without rebasing — keep history linear
- Do not skip security-reviewer on hook submissions — it is never optional
- Do not commit schema changes mixed with application code — separate commits
- Do not store credentials, API keys, or secrets in any registry file or hook implementation
- Do not use `any` in TypeScript — strict mode is enforced
- Do not access `process.env` directly in application code — use `src/config.ts` (zod-validated)
- Do not merge a PR with a 🔴 Critical finding from security-reviewer
- Do not add hooks to the registry without running `pnpm run validate-all` first
- Do not hardcode the registry URL — use the config constant in `packages/cli/src/registry/client.ts`
